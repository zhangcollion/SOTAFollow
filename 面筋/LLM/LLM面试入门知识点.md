# 一、模型架构深度理解 (Architecture)

不仅仅要懂 Transformer，更要懂它的演进和变体。

## Attention 变体

- **MHA vs. MQA vs. GQA**：重点理解 GQA (Grouped Query Attention) 如何在推理速度（KV Cache 显存占用）和模型效果之间做权衡。
- **MLA (Multi-head Latent Attention)**：DeepSeek 系列的核心，理解它如何通过低秩压缩显著降低推理时的 KV Cache 压力。
- **FlashAttention (1/2/3)**：必考。重点在于它是如何利用 GPU 内存分级（SRAM vs. HBM）来消除 IO 瓶颈，而不是改变 O(n²) 的算法复杂度。

## 位置编码 (Positional Encoding)

- **RoPE (Rotary Positional Embedding)**：深入理解其复数旋转矩阵的形式，以及为什么它在长度外推性上优于绝对位置编码。

  RoPE（旋转位置编码）是 LLaMA、Mistral、Qwen 等主流 LLM 的标准位置编码。核心是**将绝对位置信息通过旋转变换融入 Q/K 向量，让 Attention 分数只依赖相对位置**，不新增参数、计算高效。

  **数学定义**

  - 维度分组：将 Q/K 向量按**两两一组**（d 为总维度，共 d/2 组），在二维复平面做旋转

  - 基频公式：第 i 组的旋转基频， 只与token 所在维度i有关
    $$
    θ_i=10000^{(−2i/d)},i=0,1,...,d/2−1
    $$
    高频维度（i 小）：旋转快、波长短，负责相邻 token 分辨，负责细粒度、短距离的区分；低频维度（i 大）：旋转慢、波长长，负责长距离定位， 粗粒度；

  - 位置 m 旋转编码

    qm=Rm⋅q,kn=Rn⋅k

    内积结果仅与相对距离 Δ=m−n 有关。

- **外推策略**：了解 NTK-aware RoPE、YaRN、LongRoPE 等处理长上下文（如 1M+ token）的技术。

  **外推策略详细讲解**

  预训练时模型仅见过**最大长度L**（如 LLaMA-2 4k）的位置分布；当推理序列长度N>L时：

  1. 旋转角度超出训练分布，高频维度快速饱和、长距离位置区分度骤降
  2. Attention 权重异常、语义断裂、困惑度飙升、生成 “失忆”
  3. 本质：**原生 RoPE 是插值友好、外推极差**的编码

  **主流 RoPE 外推方案**

  **线性插值**：修改token 位置index m， 把长序列位置索引**按缩放因子 s 压缩**到预训练窗口内，让模型 “以为” 仍在处理训练长度，无需重训即可扩展。**优缺点**：优， 实现极简、零训练、直接推理扩展 2~8 倍；缺，统一压缩所有维度，**高频被过度压缩，破坏相邻 token 分辨、短文本性能下降**；长距离仍有混淆

  **NTK 感知插值** ：增大基频， 实现**高频维度不缩放 / 少缩放、低频维度大幅缩放**，保留局部细节、拉伸长距离定位。高频i 越小，几乎不缩放；低频i 越大，缩放越强，拉伸长距离;  **优缺点**： 优，保留高频局部精度、长距离外推显著优于 PI、无需重训；缺，极端长序列（>32k）仍有精度衰减、需配合微调

  **YaRN**： 解决了 PI 和原始 NTK 的三个痛点，**高频保留**：继承 NTK 思想，对高频维度（负责局部细节）少缩放，对低频维度（负责长程依赖）多缩放。**分段处理（NTK-by-parts）**：根据每个维度的“波长”动态决定策略，而非对所有维度一刀切。**分布修正**：长序列会导致注意力分布熵变低（过于尖锐），引入温度系数 `t`进行平滑。【没看明白。。。】

  **Sliding Window Attention**：不修改 RoPE，而是限制 Attention 只关注**最近 W 个 token**（如 W=4k/8k），超出部分忽略，强制控制 KV Cache 与计算量。**优缺点**：优，显存 / 计算可控、推理极快、无外推失效； 缺，无法捕捉**超过窗口的全局长依赖**，适合长文本摘要 / 对话，不适合全文检索

  **外推方案对比**

  | 方案           | 核心修改          | 训练需求 | 外推上限 | 短文本 | 适用场景           |
  | -------------- | ----------------- | -------- | -------- | ------ | ------------------ |
  | 原生 RoPE      | 无                | 预训练   | ≤L       | 最优   | 标准推理           |
  | 线性插值       | 缩放位置 m        | 无       | 2~8x     | 下降   | 快速小扩展         |
  | NTK-Aware      | 缩放基频θi        | 无       | 8~32x    | 良好   | 主流长上下文       |
  | YaRN           | 分维度 + 动态基频 | 极小微调 | 32~128k+ | 最优   | 超长上下文 SOTA    |
  | Sliding Window | 截断 Attention    | 无       | 窗口内   | 最优   | 低延迟、长文本对话 |

  **补充知识**：

  1、SRAM（静态随机存储器）：

  GPU **片上高速缓存**（L1/L2/Shared Memory），**不是普通显存**。速度极快（延迟 < 1ns）、带宽极高（几十 TB/s）、容量极小（MB 级），**只能存小块数据**。价格昂贵；

  **数据流向**：GPU 核心要数据时，先找 **L1** -> 找不到找 **L2** -> 再找不到才去 **HBM 显存**（最慢）

  2、HBM（高带宽存储器）：

  GPU **专用主显存**（3D 堆叠 DRAM），**平时说的 “GPU 显存** 就是它（A100/H100 的显存）。容量大（几十～百 GB）、带宽高（~2–3TB/s）、延迟几十 ns。存模型权重、**KV Cache**、激活等。
  
  华为昇腾使用HBM2e(受制于供应链和封装技术限制),  相比nvidia 落后一代(使用HBM3)
  
  3、内存（RAM / 系统内存）：CPU 用的内存（DDR5），GPU 不能直接访问，需 PCIe 拷贝，**速度慢、带宽低**。

# RoPE 原理与长上下文外推方案详解

## 一、RoPE 基础原理

### 1.1 核心概念

RoPE（Rotary Position Embedding，旋转位置编码），是目前 LLaMA、Qwen、Mistral 等大模型**通用标配位置编码**。

核心逻辑：

不额外增加位置向量，**通过旋转矩阵，把位置信息编码进 Query / Key 向量**，

最终注意力内积只和**相对位置**相关，具备天然的相对位置建模能力。

### 1.2 基础数学规则

1. 将 Q、Key 向量按**两个维度为一组**切分；
2. 对每组向量，根据当前 Token 位置，施加 2D 旋转变换；
3. 基础频率公式：

θi=10000−d2i

1. 位置 m 旋转矩阵作用：

qm=Rm⋅q,kn=Rn⋅k

1. 内积结果仅与相对距离 Δ=m−n 有关。

### 1.3 原生 RoPE 致命缺陷（外推问题）

模型预训练只有固定上下文长度（如 4k）。

当推理长度 > 训练长度：

- 旋转角度超出训练分布
- 高频维度角度饱和
- 注意力权重混乱、上下文错位、生成崩坏
- 长文本理解急剧下降

------

## 二、主流 RoPE 外推解决方案

### 2.1 PI 线性插值（Position Interpolation）

#### 核心思路

不修改频率，**直接压缩位置下标**，把超长序列强行压缩到训练长度范围内。

- 缩放系数：s=训练长度推理长度
- 新位置：pos′=spos

#### 优缺点

- 优点：零训练、实现最简单
- 缺点：全局统一压缩，**局部短句语义受损严重**，精度掉点明显

------

### 2.2 NTK-Aware RoPE（工业最常用）

#### 核心原理

不压缩位置，**修改 RoPE 基础频率**。

利用 NTK 理论：

- 高频维度：几乎不缩放，保留局部语义
- 低频维度：大幅拉伸频率，支持超长距离依赖

修正频率公式：

θi′=θi⋅sd2i

#### 优缺点

- 优点：长短文本兼顾、无需微调、落地最简单、性价比最高
- 缺点：128k 以上极限长文本仍有轻微衰减

------

### 2.3 Dynamic NTK

在 NTK 基础上**动态自适应缩放系数**：

根据当前实际生成长度，实时动态调整 s，

避免固定系数带来的长度适配僵硬问题。

------

### 2.4 YaRN（当前长文本 SOTA）

#### 核心改进

融合 PI + NTK + 分段频率修正：

1. 高频段：保持原生 RoPE 不变
2. 中频段：混合插值
3. 低频段：强 NTK 拉伸
4. 增加数值温度修正，稳定长文本注意力

#### 特点

- 只需极少量微调
- 支持 128k～256k 超长上下文
- 长短文本效果几乎无损
- 目前开源长上下文模型最优方案

------

### 2.5 滑动窗口注意力 Sliding Window

不修改 RoPE，从注意力机制限制：

每个 Token 只看**最近 N 个历史 Token**（如 4k / 8k）。

- 优点：彻底规避外推问题、KV Cache 可控、显存压力小
- 缺点：无法全局长依赖，适合对话、不适合超长文档理解
